# v0.11 Deferred Gates

This document owns the gates intentionally deferred from v0.10. A v0.10 release
may sign off with these items accepted as deferred work when the release notes
and sign-off record point here.

## Lab Validation Gates

The following gates require release-candidate packages and a lab environment
with VPP, FRR, HA peers, and realistic management traffic:

- HA failover soak.
- FRR and VPP restart recovery.
- 24-hour resource leak and churn testing.

v0.10 status:

- Runbooks are documented in `docs/v0.10-operational-runbook.md` and
  `docs/v0.10-operational-runbook.ja.md`.
- Actual lab evidence is not required for v0.10 sign-off when no lab is
  available.
- The sign-off record must list these as accepted v0.11 deferred gates.

v0.11 close criteria:

- Record package version, commit SHA, host OS, VPP version, FRR version, and
  topology.
- Attach command output before, during, and after each planned disruption.
- Attach daemon logs and Prometheus metrics covering the full test window.
- Record convergence time, any transient warnings, and final owner sign-off.
- File follow-up issues for every unexplained data-plane, control-plane, or
  management-plane failure.

## NETCONF Startup Datastore

Do not advertise `urn:ietf:params:netconf:capability:startup:1.0` until all
semantics below are implemented and tested.

### Capability Model

- Startup datastore support is opt-in for v0.11.
- When disabled, `<startup/>` RPC targets keep returning
  `operation-not-supported`, matching v0.10 behavior.
- When enabled, `<hello>` advertises `:startup` only after the datastore backend
  has completed the startup storage migration.

### Storage Model

- `startup` is a separate persisted record, not an alias for the latest running
  config.
- SQLite and etcd backends must both represent startup independently from
  running, candidate, commit history, locks, and rollback archives.
- Migration initializes startup from the latest running snapshot exactly once
  for existing deployments.
- A failed migration must leave the previous running datastore readable by the
  older binary.

### RPC Semantics

- `<get-config><source><startup/></source></get-config>` returns the startup
  record.
- `<validate><source><startup/></source></validate>` parses and validates the
  startup record without applying it.
- `<copy-config>` supports:
  - `running` to `startup`
  - `candidate` to `startup`
  - inline `config` to `startup`
  - `startup` to `candidate`
- `<copy-config>` from `startup` to `running` stays unsupported unless the
  server also implements and advertises `:writable-running`.
- `<edit-config target=startup>` stays unsupported for the first v0.11
  implementation. Operators should use `copy-config` to replace startup
  atomically.
- `<delete-config target=startup>` resets startup to the default empty config.

### Locking and Boot Behavior

- Startup uses an independent lock namespace.
- Candidate locks must not block startup read operations.
- Startup locks must block startup writes and deletes, but must not block
  running commits.
- Daemon startup loads and applies the startup record when the feature is
  enabled. Running state is then initialized from the applied startup config.
- If startup validation fails at daemon boot, the daemon must fail closed and
  report an actionable error instead of silently falling back to stale running
  state.

### Test Gates

- SQLite and etcd migration tests.
- RPC tests for get-config, validate, copy-config, delete-config, lock, and
  unlock.
- Negative tests proving v0.10 behavior remains unchanged when startup is
  disabled.
- Upgrade/downgrade guard tests for schema compatibility.
- External client smoke tests for the advertised `:startup` capability.

## Standard NETCONF `:xpath` Default Promotion

v0.10 includes an explicit opt-in path for
`urn:ietf:params:netconf:capability:xpath:1.0`. It is enabled with
`arca-routerd --netconf-standard-xpath` or
`tools/netconf-interop-server -standard-xpath`, and it must not be documented
as supported unless standard XPath evidence is attached to sign-off.

The v0.11 follow-up is only default promotion:

- Keep default `<hello>` behavior conservative unless release-candidate
  operators accept the evidence.
- Before changing the default, attach standard XPath evidence from ncclient and
  libnetconf2-family clients.
- Preserve the same DoS boundaries for expression length, XML input size,
  selected element count, output size, depth, attribute count, and evaluation
  timeout.
- Record client version, RPC payload, server `<hello>`, reply XML, and any
  interoperability deviations before changing the default.

Use `docs/netconf-xpath-interop.md` as the executable runbook.

## v0.10 Sign-off Use

When signing off v0.10 without these gates:

- List each deferred gate in the sign-off record.
- Explain why it is safe to defer from v0.10.
- Confirm `arca show compatibility` and `docs/compatibility.md` do not claim
  support for startup datastore or completed lab soak. Standard `:xpath` may be
  claimed only when the standard XPath evidence is linked.
- Open or reference v0.11 tracking issues before tagging v0.10.
