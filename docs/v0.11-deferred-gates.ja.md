# v0.11 Deferred Gates

この文書は、v0.10 から意図的に defer する gate の source of truth とする。v0.10 release は、release notes と sign-off record がこの文書を参照し、deferred work として明示している場合に限り、これらを未完了のまま sign-off できる。

## Lab Validation Gates

以下の gate は、VPP、FRR、HA peer、現実的な management traffic を含む lab 環境と release-candidate package が必要。

- HA failover soak。
- FRR / VPP restart recovery。
- 24-hour resource leak / churn testing。

v0.10 status:

- 手順は `docs/v0.10-operational-runbook.md` と `docs/v0.10-operational-runbook.ja.md` に文書化済み。
- lab が利用できない場合、実際の lab evidence は v0.10 sign-off の必須条件にしない。
- sign-off record には、これらを accepted v0.11 deferred gate として記録する。

v0.11 close criteria:

- package version、commit SHA、host OS、VPP version、FRR version、topology を記録する。
- planned disruption の前、最中、後の command output を添付する。
- test window 全体を覆う daemon log と Prometheus metrics を添付する。
- convergence time、transient warning、final owner sign-off を記録する。
- data-plane、control-plane、management-plane の説明できない failure は follow-up issue を作成する。

## NETCONF Startup Datastore

以下の semantics が実装および test 済みになるまで、`urn:ietf:params:netconf:capability:startup:1.0` を advertise しない。

### Capability Model

- startup datastore support は v0.11 では opt-in とする。
- disabled の場合、`<startup/>` RPC target は v0.10 と同じく `operation-not-supported` を返す。
- enabled の場合、datastore backend の startup storage migration 完了後にのみ `<hello>` で `:startup` を advertise する。

### Storage Model

- `startup` は latest running config の alias ではなく、独立した persisted record とする。
- SQLite と etcd backend は、startup を running、candidate、commit history、lock、rollback archive から独立して表現する。
- migration は既存 deployment に対して latest running snapshot から startup を 1 回だけ初期化する。
- migration が失敗した場合、古い binary が従来の running datastore を読める状態を維持する。

### RPC Semantics

- `<get-config><source><startup/></source></get-config>` は startup record を返す。
- `<validate><source><startup/></source></validate>` は startup record を apply せず parse / validate する。
- `<copy-config>` は以下を support する。
  - `running` to `startup`
  - `candidate` to `startup`
  - inline `config` to `startup`
  - `startup` to `candidate`
- `startup` to `running` の `<copy-config>` は、server が `:writable-running` も実装・advertise しない限り unsupported のままとする。
- 初回 v0.11 実装では `<edit-config target=startup>` は unsupported のままとする。operator は `copy-config` により startup 全体を atomic に replace する。
- `<delete-config target=startup>` は startup を default empty config に reset する。

### Locking and Boot Behavior

- startup は独立した lock namespace を使う。
- candidate lock は startup read operation を block しない。
- startup lock は startup write/delete を block するが、running commit は block しない。
- feature enabled の場合、daemon startup は startup record を load / apply し、その適用結果から running state を初期化する。
- boot 時に startup validation が失敗した場合、daemon は stale running state に silently fallback せず、actionable error を出して fail closed する。

### Test Gates

- SQLite / etcd migration test。
- get-config、validate、copy-config、delete-config、lock、unlock の RPC test。
- startup disabled 時に v0.10 behavior が変わらないことを示す negative test。
- schema compatibility の upgrade / downgrade guard test。
- advertised `:startup` capability に対する external client smoke test。

## Standard NETCONF `:xpath`

implementation が Arca subset だけではなく standard NETCONF XPath として動作するまで、`urn:ietf:params:netconf:capability:xpath:1.0` を advertise しない。

v0.10 status:

- Arca は implementation-specific な XPath filter subset を support できる。
- standard `:xpath` は advertise しない。

v0.11 close criteria:

- `get` と `get-config` について RFC 6241 response behavior を cover する。
- XPath evaluation は node-set を返し、scalar result は `invalid-value` として reject する。
- namespace prefix を RPC、filter、selected node の level で test する。
- empty result set は empty data の successful reply を返す。
- invalid XPath、undeclared prefix、unsupported node type、unsupported path は deterministic な `rpc-error` を返す。
- expression length、XML input size、selected element count、output size、depth、attribute count、evaluation timeout の limit を cover する。

external validation には、Arca の test helper を共有しない client family を 2 種類以上使う。

- `ncclient-family`: baseline は `ncclient`。Junos-style automation の確認が必要な場合、PyEZ を supplementary ncclient-family smoke として追加できる。
- `libnetconf2-family`: Netopeer2 `netopeer2-cli` またはその他の libnetconf2-based client。standard capability を advertise する前に、この family の evidence が必須。

`ncclient` がすでに pass している場合、PyEZ は 2 種類目の independent-family requirement を満たさない。`netconf-console` は、導入されている tool が ncclient-backed ではないことを確認できる場合のみ acceptable とする。

standard capability を enable する前に、client version、RPC payload、server `<hello>`、reply XML、interoperability deviation を記録する。

この gate の executable runbook として `docs/netconf-xpath-interop.md` を使う。

## v0.10 Sign-off Use

これらの gate なしで v0.10 を sign-off する場合:

- sign-off record に各 deferred gate を列挙する。
- v0.10 から defer して安全な理由を説明する。
- `arca show compatibility` と `docs/compatibility.md` が startup datastore、standard `:xpath`、完了済み lab soak を support として主張していないことを確認する。
- v0.10 tag 作成前に v0.11 tracking issue を作成または参照する。
