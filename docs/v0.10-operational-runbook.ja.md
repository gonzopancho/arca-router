# v0.10 運用 Runbook

この runbook は、v0.10 stabilization 期間に残っている手動検証と復旧確認を定義する。リリース候補を ready と判断する前に、各セクションのコマンド出力、時刻、daemon log、package version を記録すること。

## Upgrade 前 baseline

package を置き換える前に、以下を実行する。

```bash
arca show compatibility
arca check upgrade
arca check upgrade backup /var/backups/arca-router/pre-upgrade.conf
arca show ha
arca show lcp
arca show class-of-service
arca show telemetry paths
systemctl status arca-routerd
```

期待結果:

- `arca check upgrade` に warning がない。warning がある場合は、すべてに operator 判断が記録されている。
- rollback archive に有効な entry が 1 件以上ある。
- configuration backup path が書き込み可能で、backup file がまだ存在しない。
- HA、LCP、QoS の出力に、導入済み機能セットに対する想定外の error がない。

## Management Security Smoke

remote management を使う deployment では、gRPC と Web/NMS API の security path を両方確認する。

```bash
make security-audit
```

```bash
arca-routerd \
  --grpc-listen=127.0.0.1:9443 \
  --grpc-tls-cert=/etc/arca-router/tls/server.crt \
  --grpc-tls-key=/etc/arca-router/tls/server.key \
  --grpc-client-ca=/etc/arca-router/tls/client-ca.crt

arca \
  -grpc-address=127.0.0.1:9443 \
  -grpc-ca=/etc/arca-router/tls/ca.crt \
  -grpc-server-name=arca-routerd.example.net \
  -grpc-client-cert=/etc/arca-router/tls/client.crt \
  -grpc-client-key=/etc/arca-router/tls/client.key \
  show compatibility
```

```bash
install -m 0600 /dev/null /etc/arca-router/web-api-tokens
printf 'nms:read-only:%s\n' "${NMS_TOKEN}" > /etc/arca-router/web-api-tokens
systemctl restart arca-routerd
curl -fsS -H "Authorization: Bearer ${NMS_TOKEN}" \
  http://127.0.0.1:8080/api/nms/v1/status
```

期待結果:

- installed service user、capabilities、config file mode、VPP socket、FRR
  file-backend readiness が `docs/security-model.md` と一致する。
- mTLS が有効な場合、信頼済み client certificate がない gRPC 接続は失敗する。
- `0600` より広い権限の Web/NMS token file は daemon startup 時に拒否される。
- `read-only` token は write endpoint を呼べない。意図した場合のみ、`operator` または `admin` token が config validate / commit を実行できる。

## HA Failover Soak

VRRP、etcd config sync、VPP LCP、FRR が有効な clustered deployment で実行する。

1. baseline 出力を取得する。

   ```bash
   arca show ha
   arca show vrrp
   arca show lcp
   curl -fsS http://127.0.0.1:9090/metrics > /tmp/arca-metrics-before.txt
   ```

2. lab plan に従って、active node の `arca-routerd` 停止または management plane 隔離により failover させる。
3. 5 秒ごとに 10 分以上 poll する。

   ```bash
   arca show ha
   arca show vrrp
   arca show lcp
   ```

4. failed node を復旧し、さらに 10 分以上 poll を継続する。

期待結果:

- convergence 後、各 VRRP group で active node がちょうど 1 台になる。
- 計画された disruption window 後、`arca show ha` が converged status を返す。
- failed node 再参加後、stale な config sync error が残らない。
- VPP LCP reconciliation に永続的な inconsistency が残らない。

## FRR / VPP Restart Recovery

可能であれば traffic generator または protocol neighbor を用意した状態で restart recovery を確認する。

```bash
systemctl restart frr
arca show bgp summary
arca show ospf neighbor
arca show vrrp
arca show bfd status

systemctl restart vpp
systemctl restart arca-routerd
arca show interfaces
arca show lcp
arca show routes
```

期待結果:

- FRR のみ restart した後、daemon restart なしで FRR status command が復旧する。
- VPP restart 後、`arca-routerd` を restart すると managed interface、LCP、route、QoS intent state が復元される。
- BGP、OSPF、VRRP、BFD session が release candidate の documented target 以内に収束する。

## Datastore Lock Recovery

daemon startup が stale ephemeral state を削除し、running configuration を破損しないことを確認する。

```bash
systemctl stop arca-routerd
cp /var/lib/arca-router/config.db /var/backups/arca-router/config.db.pre-lock-test
systemctl start arca-routerd
arca check upgrade
arca show configuration
```

SQLite failure injection を lab で行う場合は、test build または SQLite client で candidate lock と candidate text を作成し、lock を解放しないまま daemon を停止してから `arca-routerd` を restart する。

期待結果:

- startup により stale candidate lock と candidate text が削除される。
- running configuration と commit history は維持される。
- cleanup 後に新しい commit を prepare / commit できる。
- `arca check upgrade` が引き続き有効な rollback archive を報告する。

## Resource Leak / Churn

release-candidate package 上で 24 時間以上 churn test を実行する。

```bash
while true; do
  arca show interfaces >/dev/null 2>&1 || true
  arca show telemetry path /system count 1 >/dev/null 2>&1 || true
  arca show telemetry path /interfaces count 1 >/dev/null 2>&1 || true
  curl -fsS http://127.0.0.1:8080/api/status >/dev/null || true
  sleep 5
done
```

以下を収集する。

```bash
systemctl status arca-routerd
journalctl -u arca-routerd --since '24 hours ago'
curl -fsS http://127.0.0.1:9090/metrics
ps -o pid,rss,vsz,etime,comm -C arca-routerd
```

期待結果:

- warm-up 後、memory growth が安定する。
- file descriptor または socket の増加が polling cycle をまたいで継続しない。
- Web/NMS、CLI、NETCONF、telemetry request が成功し続ける。

## Upgrade Rollback

upgrade 後の package が startup または post-upgrade validation に失敗した場合:

1. 以前の package artifact を reinstall する、または以前の package repository pin を復元する。
2. `arca-routerd` を restart する。
3. running config が変更された場合のみ、pre-upgrade configuration を復旧する。

   ```bash
   arca show configuration rollback 1
   arca backup configuration rollback 1 /var/backups/arca-router/rollback-1.conf
   ```

4. `arca check upgrade` を再実行し、失敗した package version、log、rollback evidence を記録する。

期待結果:

- 以前の package が保存済み datastore で起動する。
- operator が rollback archive configuration を確認し、export できる。
- config restore は明示的に行われ、commit 前に validate される。
