# v0.10 Release Readiness Checklist

This checklist records the remaining release-readiness gates for the v0.10
stabilization line. Treat `docs/compatibility.md`, `internal/compat/policy.go`,
and `arca show compatibility` as the source of truth for compatibility and
support matrix content.

## Documentation Freeze

Before cutting a release candidate, freeze the following documents and confirm
that any post-freeze change is listed in the release notes:

- `README.md`
- `README.ja.md`
- `CHANGELOG.md`
- `ROADMAP.md`
- `docs/compatibility.md`
- `docs/observability.md`
- `docs/security-model.md`
- `docs/v0.10-operational-runbook.md`
- `docs/v0.10-operational-runbook.ja.md`
- `docs/v0.10-release-signoff.md`
- `docs/v0.10-release-signoff.ja.md`
- `docs/v0.11-deferred-gates.md`
- `docs/v0.11-deferred-gates.ja.md`
- `docs/release-process.md`

Freeze criteria:

- CLI commands in docs match `arca -help` and interactive help.
- Compatibility schema IDs match `internal/compat/policy.go`.
- Management security options match daemon and CLI flags.
- Upgrade rollback guidance matches `arca check upgrade` output.
- Operational runbook commands have a recorded owner for release-candidate
  execution, or a v0.11 deferred owner when lab execution is unavailable.

## Support Matrix

The v0.10 support matrix is published through `arca show compatibility` and
`docs/compatibility.md`.

| Component | Supported | Required | Release Gate |
| --- | --- | --- | --- |
| VPP | 24.10+ | `vpp`, `vpp-plugin-core`, linux-cp plugin | Verify installed package version and LCP/QoS capability output; defer restart/soak evidence to v0.11 when lab access is unavailable. |
| FRR | 8.0+ | `bgpd`, `ospfd`, `ospf6d`, `zebra`, `staticd`, `mgmtd`, `vrrpd`, `bfdd` | Verify package and status command availability; defer restart recovery evidence to v0.11 when lab access is unavailable. |
| SQLite datastore | schema 1-2 | current schema 2 | Run `arca check upgrade`; newer schema versions must be rejected by older binaries. |
| NETCONF | base:1.0 and base:1.1 | candidate, validate, rollback-on-error; standard `:xpath` opt-in | Confirm startup datastore remains unadvertised. Standard `:xpath` may be advertised only when `--netconf-standard-xpath` is enabled and standard XPath evidence is linked. |
| gRPC | `arca.router.v1` | Unix socket by default; optional TCP/TLS/mTLS | Verify local socket access and remote TLS/mTLS smoke checks. |
| NMS JSON | `arca.nms.*.v1` | additive schemas | Verify status, telemetry catalog, schemas, and snapshot endpoints with the NMS collector example. |

## Release Candidate Evidence

Attach or link the following evidence before marking v0.10 release-ready:

- `go test ./...`
- `go vet ./...`
- `make package-lint`
- `make release-check` output when using the aggregate local readiness check.
- `make security-audit` output from the installed release-candidate host when
  that host is available; otherwise record the missing host as accepted
  deferred evidence.
- Package build metadata for DEB/RPM artifacts.
- `arca show compatibility` output from the release-candidate package.
- `arca check upgrade` output from the release-candidate package.
- `NETCONF Client Interoperability` artifacts for ncclient and libnetconf2
  default-mode runs, proving that standard `:xpath` remains unadvertised while
  the Arca XPath subset interops with both client families.
- `netconf-client-ncclient-standard-xpath-evidence` and
  `netconf-client-libnetconf2-standard-xpath-evidence`, or verified local
  `make netconf-standard-xpath-evidence` /
  `make netconf-standard-xpath-evidence-verify` output, before documenting or
  enabling standard `:xpath` support for v0.10.
- Completed applicable sections of `docs/v0.10-operational-runbook.md` or
  `docs/v0.10-operational-runbook.ja.md`.
- Any accepted warnings or deferred gates, with owner and rationale.

## v0.11 Deferred Gates

The following items are intentionally deferred to v0.11. Do not represent them
as v0.10 release evidence unless a lab run is completed and attached.

- HA failover soak, FRR/VPP restart recovery, and 24-hour churn results for
  release-candidate packages.
- Formal NETCONF startup datastore support.

Use `docs/v0.11-deferred-gates.md` as the source of truth for entry criteria,
implementation semantics, and evidence required to close these gates.

## Release Sign-off

Record release sign-off in the release PR, issue, or tag notes before creating
the final v0.10 tag. Use `docs/v0.10-release-signoff.md` as the template. The
sign-off record must include:

- Sign-off owner.
- Date and timezone.
- Commit SHA or release tag.
- Evidence links for tests, package metadata, compatibility output, and
  applicable runbook sections.
- Accepted warnings and v0.11 deferred gates.
- Rationale for each deferred gate.
